Manual Técnico Completo · Streaming NDI LAN

Guía integral del emisor NDI por red local: arquitectura, flujo interno, decisiones de diseño, operación, diagnóstico y extensión.

Proyecto: Py_NDI Enfoque: streaming NDI LAN Última actualización: 2026-04-21

1. Resumen Ejecutivo

El proyecto publica una señal NDI a partir de un archivo de vídeo local, con audio sincronizado y modo dual opcional para validación visual (señal secundaria con overlay).

Estado: operativo Video + audio Modo dual opcional Pacing monotónico

  • Punto de entrada: stream_video.py.
  • Módulo auxiliar: utils.py.
  • Dependencias Python: cyndilib y numpy.
  • Herramientas externas: ffmpeg y ffprobe.

3. Stack Técnico

ComponenteUso realMotivo

4. Arquitectura del Pipeline

CLI -> parse args -> ffprobe metadata -> sender config (NDI)
    -> ffmpeg video raw BGRA pipe + ffmpeg audio f32le
    -> bucle:
         frame BGRA
         bloque audio (channels,samples)
         write_video_and_audio()
         pacing monotónico

Contratos de datos

  • Video: formato BGRA, 8 bits por canal, buffer lineal.
  • Frame bytes: width * height * 4.
  • Audio: float32, shape (channels, samples).
  • Sample rate: 48 kHz, 2 canales.

5. Flujo de Ejecución Paso a Paso

  1. Se parsean argumentos y se selecciona vídeo de entrada (por defecto en Videos/).
  2. Se consulta metadata con ffprobe para ancho, alto, fps y frames.
  3. Se crea y configura el sender NDI principal.
  4. Se calcula bloque de audio por frame según fps declarado.
  5. Se decodifica audio completo a memoria para acceso determinista.
  6. Se arranca decoder de vídeo por pipe en formato rawvideo BGRA.
  7. En cada frame se lee vídeo, se toma bloque de audio y se envía.
  8. Si el pipe falla, se reinicia decoder y se resincroniza cursor de audio.
  9. Un reloj monotónico acumulativo controla el pacing.

8. Temporización y Sincronización

8.1 Pacing de vídeo

  • Se usa next_frame_time += frame_duration para scheduling acumulativo.
  • Si el loop va tarde, se re-sincroniza con now para no arrastrar drift indefinido.

8.2 Relación audio/frame

  • audio_samples_per_frame = round(sample_rate / fps).
  • Si la relación es fraccional, se emite warning y se prioriza bloque fijo para estabilidad.

8.3 Integridad de buffer

  • Se usa np.frombuffer(...).copy() para garantizar memoria escribible en envío NDI.
  • Evita el error: ValueError: buffer source array is read-only.

9. Modo Dual

Con --dual se crea una segunda fuente NDI con el sufijo -Square.

  • La señal principal mantiene vídeo limpio.
  • La secundaria aplica overlay animado por frame.
  • Ambas pueden compartir el mismo bloque de audio por frame.

10. Ejecución y Operativa

10.1 Instalación

cd <project-root>
python3 -m pip install -r requirements.txt

10.2 Ejecución

python3 ./stream_video.py
python3 ./stream_video.py Videos/alhaja.mp4
python3 ./stream_video.py Videos/alhaja.mp4 --dual

10.3 Verificación rápida

  1. Abrir NDI Monitor o receptor compatible.
  2. Confirmar visibilidad de fuente principal.
  3. En dual, confirmar fuente secundaria con overlay.
  4. Comprobar audio en ambas fuentes si el archivo lo contiene.

11. Troubleshooting Avanzado

SíntomaCausa probableAcción recomendada
No hay salida NDI Runtime NDI no operativo o red fuera de alcance. Verificar instalación runtime y segmento LAN compartido.
No hay audio ffmpeg ausente o archivo sin pista de audio. Instalar ffmpeg y validar pista con ffprobe.
Jitter o stutter Sobrecarga CPU, overruns de timing o archivo muy pesado. Reducir resolución/fps del asset y validar carga del host.
Error read-only buffer Frame generado desde bytes no escribibles. Usar copia writable antes de envío (ya integrado).
Fallos intermitentes de frame Pipe de decoder cerrado inesperadamente. El script reinicia decoder automáticamente y continúa.

12. Rendimiento y Escalabilidad

  • La ruta crítica está dominada por decode + envío NDI por frame.
  • Duplicar señal (modo dual) incrementa coste por copia/overlay/envío.
  • Para estabilidad en hosts limitados, priorizar assets con bitrate/fps moderado.

Recomendaciones

  • Usar archivos con fps consistente (p.ej. 50 o 60 estables).
  • Mantener software de NDI y sistema actualizados.
  • Monitorizar warnings de timing en consola durante sesiones largas.

13. Guía de Extensión

  1. Para nuevos overlays, ampliar draw_square() o crear nuevas funciones en utils.py.
  2. Para nuevos modos de salida, extender parseo CLI en el bloque __main__.
  3. Para telemetría, añadir métricas de loop y emitirse cada N frames.
  4. Para múltiples fuentes, instanciar varios senders con clocks equivalentes.

16. Conclusión Técnica

La implementación actual ofrece un pipeline NDI LAN sólido, legible y mantenible, con sincronización práctica entre vídeo y audio y mecanismos de recuperación frente a cortes del decoder.

La documentación está pensada para que cualquier persona pueda entender el sistema, operarlo y ampliarlo con bajo riesgo.

17. Decisión de Arquitectura: Migración a FFmpeg

Contexto técnico

  • La implementación original era funcional, pero en determinados entornos macOS apareció un conflicto de librerías nativas cargadas en el mismo proceso.
  • OpenCV cargaba una variante de libavdevice y el stack NDI HX cargaba otra variante con clases Objective-C duplicadas.
  • Ese escenario podía producir errores de casting espurios, comportamiento no determinista y riesgo de crash.

Decisión tomada

  • Eliminar OpenCV del camino de runtime del streamer.
  • Migrar probing/decoding a ffprobe/ffmpeg.
  • Mantener el procesamiento de frame y overlay en numpy.

Beneficios obtenidos

  • Eliminación de la causa raíz del conflicto nativo en el proceso de emisión.
  • Pipeline más predecible para integración inter-organizacional.
  • Compatibilidad mantenida con flujos basados en np.ndarray, incluyendo etapas posteriores de procesamiento.
  • Rendimiento de decode en línea con los objetivos del proyecto (normalmente comparable o mejor, según host y carga).

Alcance de validación

La solución ha sido testada en macOS y Apple Vision Pro.
